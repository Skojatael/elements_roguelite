# Implementation Plan: Enemy Spawn Delay

**Branch**: `057-enemy-spawn-delay` | **Date**: 2026-03-18 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/057-enemy-spawn-delay/spec.md`

## Summary

Every enemy receives a random 1–2 second inactivity window immediately after spawning. During this window, the enemy does not move or deal contact damage. After the window expires, all normal behaviour resumes. The change is confined to `scenes/combat/enemies/Enemy.gd` — a per-instance float field and a delta countdown guard in `_physics_process`.

## Technical Context

**Language/Version**: GDScript 4.6 (static typing)
**Primary Dependencies**: None — modifies existing Enemy node only
**Storage**: N/A
**Testing**: GUT (headless) — pure logic not extractable; manual in-editor validation only
**Target Platform**: Android mobile / Windows dev
**Performance Goals**: O(1) per enemy per frame — single float comparison, no allocations
**Constraints**: Must not affect enemy damage-taking or death during delay
**Scale/Scope**: Applies to all enemy instances across all room types

## Constitution Check

- **I. Single Responsibility** ✅ — All changes live in `Enemy.gd`, which already owns enemy AI behaviour. No new files, no cross-script wiring.
- **II. Data-Driven Content** ✅ — `spawn_delay` stored in `data/dungeon_config.json` under a top-level `"enemy_spawn"` block. `Enemy.gd` reads the value via `ResourceManager.get_dungeon_config()`.
- **III. Mobile-First Performance** ✅ — One float decrement + one comparison added to `_physics_process`. Negligible cost at any enemy count.
- **IV. Editor-Centric Workflow** ✅ — No `.tscn` changes. No new node references.
- **V. Simplicity & YAGNI** ✅ — No new abstraction. Single field, single guard. No placeholder hooks.
- **VI. Early Return** ✅ — Delay guard will use `if _spawn_delay > 0.0: _spawn_delay -= delta; return` at the top of `_physics_process`, keeping the happy path unindented.
