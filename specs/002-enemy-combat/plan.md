# Implementation Plan: Enemy Combat System

**Feature Directory**: `002-enemy-combat` | **Date**: 2026-02-19 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/002-enemy-combat/spec.md`

---

## Summary

Add a functional enemy combat system to the dungeon. Enemies are CharacterBody2D scenes that load stats from `data/enemies.json` via typed `EnemyData` wrappers, pursue the player on detection, deal contact damage, and die when health is depleted. The player's `StatsComponent` tracks health and triggers `GlobalSignals.gameplay_ended` on death. The player's `CombatComponent` auto-attacks overlapping enemies on a timer.

---

## Technical Context

**Language/Version**: GDScript 4.6 (static typing required)
**Primary Dependencies**: Godot 4.6 built-ins — `CharacterBody2D`, `Area2D`, `CollisionShape2D`, signals, `JSON`
**Storage**: `data/enemies.json` (read at runtime; no writes)
**Testing**: Manual in-editor playtesting per `quickstart.md`
**Target Platform**: Android portrait 1080×1920 (development on Windows/D3D12)
**Project Type**: Godot 4.6 scene-component game
**Performance Goals**: 60 fps with 10+ simultaneous enemies on mid-range Android (Snapdragon 665-class)
**Constraints**: Mobile renderer only (no Forward+); Jolt physics; static typing on all variables/parameters/returns
**Scale/Scope**: 1 enemy type (Slime) for MVP; data schema supports N types

## Constitution Check

*GATE: All five principles evaluated. Must pass before implementation.*

| Principle | Status | Notes |
|---|---|---|
| I. Single Responsibility | ✅ PASS | `Enemy.gd` handles one entity's behaviour (health + state + movement + contact damage are all enemy-game-logic, no mixing of data/UI/logic categories). `StatsComponent` owns player health only. `CombatComponent` owns outgoing player attacks only. No script serves two domains. |
| II. Data-Driven Content | ✅ PASS | All enemy stats (`max_health`, `damage`, `move_speed`, `detection_range`, `damage_cooldown`) live in `data/enemies.json`. `EnemyData.gd` provides a typed wrapper. No numeric constants hardcoded in `.gd` scripts. |
| III. Mobile-First Performance | ✅ PASS | Area2D signals used (event-driven, not per-frame polling). `_physics_process` movement only runs when `_state == PURSUING`. 10-enemy performance verified in Step 6 of quickstart. No Forward+ shaders. |
| IV. Editor-Centric Workflow | ✅ PASS | `Enemy.tscn` and all player component scenes built in the Godot Editor. No raw `.tscn` text edits. |
| V. Simplicity & YAGNI | ✅ PASS | Enemy uses a single co-located script (no component hierarchy until a second concrete call site exists). Auto-attack is the simplest player attack that satisfies all user stories. No speculative abstractions. |

*Post-design re-check: Constitution compliance confirmed — all decisions documented in `research.md` justify the approach against principles.*

## Key Design Decisions (summary — full rationale in `research.md`)

1. **Single `Enemy.gd`** — not a component hierarchy; YAGNI compliant; player-component mandate is player-specific per constitution.
2. **Auto-attack in `CombatComponent`** — simplest player damage delivery; no gesture system needed.
3. **Two Area2D nodes per enemy** — `DetectionArea` (pursuit toggle) + `ContactArea` (damage timer); event-driven, not polled.
4. **`StatsComponent.died` → `Main.gd` → `GlobalSignals.gameplay_ended`** — coordinator pattern; no new GlobalSignals signals needed.
5. **Flat JSON array** — `{"enemies": [...]}` in `enemies.json`; `EnemyData.from_dict()` typed constructor.
