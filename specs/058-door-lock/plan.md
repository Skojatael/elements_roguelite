# Implementation Plan: Door Lock During Combat

**Branch**: `058-door-lock` | **Date**: 2026-03-18 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/058-door-lock/spec.md`

## Summary

When enemies spawn in a combat room, `RoomSpawner` sets `locked = true` on all sibling `Door` nodes. While locked, `Door` suppresses `door_activated` so the player cannot trigger a room transition. When `room_cleared` fires, `RoomSpawner` sets `locked = false` on all doors. Rooms with no enemies (empty spawn config, already-cleared rooms) never lock. Changes touch two files: `Door.gd` (add `locked` field) and `RoomSpawner.gd` (lock/unlock calls).

## Technical Context

**Language/Version**: GDScript 4.6 (static typing)
**Primary Dependencies**: None — modifies two existing scripts
**Storage**: N/A
**Testing**: GUT — `Door` has no pure logic to unit test. `RoomSpawner` changes involve scene tree access. Manual in-editor validation only.
**Target Platform**: Android mobile / Windows dev
**Performance Goals**: O(n) over door count per room (max 4) on spawn/clear — negligible
**Constraints**: Must not affect boss room (doors already suppressed there separately); must not lock already-cleared rooms on re-entry

## Constitution Check

- **I. Single Responsibility** ✅ — `Door.gd` gains only a lock state + guard. `RoomSpawner.gd` already owns spawn lifecycle — calling lock/unlock on doors is a natural extension of that responsibility. No new files needed.
- **II. Data-Driven Content** ✅ — No new numeric constants. Lock state is a boolean derived from game events, not a config value.
- **III. Mobile-First Performance** ✅ — Lock/unlock runs once on spawn and once on clear. Zero per-frame cost.
- **IV. Editor-Centric Workflow** ✅ — No `.tscn` edits. `Door` nodes are found at runtime via `get_parent().get_children()` filtered by `is Door` (same pattern used in boss room suppression).
- **V. Simplicity & YAGNI** ✅ — `locked: bool` on `Door`, two calls in `RoomSpawner`. No new abstractions.
- **VI. Early Return** ✅ — `_on_body_entered` gains `if locked: return` at the top.

## Physical Blocking Note

`Door.tscn` is an `Area2D` — it detects player entry but does not provide physics collision. The room's boundary walls already prevent the player from leaving; the door zone sits at the wall threshold. "Locking" therefore means suppressing `door_activated` (preventing room transition), not adding a new physics body. This satisfies the spec intent: a locked door is one the player cannot pass through in a meaningful sense (no room transition fires).
