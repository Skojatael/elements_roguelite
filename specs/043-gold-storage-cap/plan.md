# Implementation Plan: Gold Offline Storage Cap

**Branch**: `043-gold-storage-cap` | **Date**: 2026-03-16 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/043-gold-storage-cap/spec.md`

## Summary

Add a configurable cap on offline gold accumulation (base: 4 hours). After the cap is exceeded no additional gold is credited. The cap is upgradeable via a new Alchemy Lab upgrade (`gold_storage_cap`). All values are data-driven in `meta_config.json`. The offline timer resets to the current time on every game open. In-session gold is unaffected.

The core change is a 2-line modification to `MetaManagerImpl.apply_offline_gold()`: clamp `elapsed` with `mini(elapsed, cap_seconds)` and update `gold_last_saved_timestamp = now_unix` after crediting. A new `gold_storage_cap_level` field in MetaState and its persistence complete the feature. UI surfaces in `LabUpgradeScreen` and `GoldDisplay`.

## Technical Context

**Language/Version**: GDScript 4.6 (static typing)
**Primary Dependencies**: Godot 4.6 Engine — no new libraries
**Storage**: `user://meta_save.json` (JSON, existing SaveManager pattern)
**Testing**: GUT 9.6.0 headless (`tests/unit/`)
**Target Platform**: Android mobile (portrait 1080×1920); Windows dev
**Performance Goals**: 60 fps — no new `_process()` work; cap is computed once per launch
**Constraints**: Constitution I–VI; no hard-coded balance values; no raw `.tscn` edits

## Constitution Check

- **I. Single Responsibility** ✅ Cap logic lives entirely in `MetaManagerImpl.apply_offline_gold()`; autoload remains a thin wrapper; `LabUpgradeScreen` gains one new button and one coordinator method.
- **II. Data-Driven Content** ✅ All cap values (`base_hours`, `hours_per_level`, `base_cost`, `cost_scale`, `max_levels`) live in `data/meta_config.json` under `alchemy_lab.upgrades.gold_storage_cap`. No magic numbers in `.gd` files.
- **III. Mobile-First Performance** ✅ Cap computation runs once in `_ready()`, not per-frame. No shaders or physics changes.
- **IV. Editor-Centric Workflow** ✅ New buttons added via Editor only; `@export var` used for all new node references; no raw `.tscn` edits.
- **V. Simplicity & YAGNI** ✅ No new abstractions. `purchase_gold_storage_cap()` reuses the existing `get_upgrade_cost()` helper (2nd call site; already used by `purchase_damage_upgrade()`). No placeholder hooks.
- **VI. Early Return** ✅ `apply_offline_gold()` retains all existing guard clauses; new cap clamping is a single-line addition after guards pass.

## Implementation Steps

### Step 1 — Config (data-first, unblocks all other steps)

Add `gold_storage_cap` to `data/meta_config.json`:

```json
"gold_storage_cap": {
  "name": "Gold Storage",
  "base_hours": 4,
  "hours_per_level": 4,
  "base_cost": 100,
  "cost_scale": 1.5,
  "max_levels": 2
}
```

### Step 2 — Data model + persistence (parallel with Step 1)

- `MetaState.gd`: add `var gold_storage_cap_level: int = 0`
- `SaveManager.gd`: add `"gold_storage_cap_level"` to save dict and `int(parsed.get(..., 0))` to load

### Step 3 — MetaManagerImpl logic (depends on Steps 1+2)

Three changes to `scripts/managers/MetaManager.gd`:

1. **Update `apply_offline_gold()` signature** — add `cap_seconds: int` parameter before `save_manager`. Update body:
   - After `elapsed > 0` guard: `var capped_elapsed: int = mini(elapsed, cap_seconds)`
   - Use `capped_elapsed` in the gold calculation (not `elapsed`)
   - Add `meta_state.gold_last_saved_timestamp = now_unix` before `_save()`
   - On first-boot branch (timestamp == 0): set timestamp to `now_unix` and save before returning

2. **Add `get_gold_storage_cap_seconds(base_hours: int, hours_per_level: int) -> int`**

3. **Add `purchase_gold_storage_cap(cost: int, max_levels: int, save_manager: Node) -> bool`** — same pattern as `purchase_damage_upgrade()`

### Step 4 — Autoload wrapper (depends on Step 3)

In `autoload/MetaManager.gd`:
- Add `gold_storage_cap_hours: int` computed property (reads config, delegates to `_impl.get_gold_storage_cap_seconds()` then divides by 3600 — or just compute directly from config + level)
- Add `purchase_gold_storage_cap() -> bool` delegation
- Update `_ready()`: compute `cap_seconds` from config and pass to `apply_offline_gold()`

### Step 5 — LabUpgradeScreen (depends on Step 4)

In `scenes/hub/LabUpgradeScreen.gd`:
- Add `@export var _storage_cap_button: Button`
- Wire `_storage_cap_button.pressed.connect(_on_storage_cap_pressed)` in `_ready()`
- Add `_update_storage_cap_button()` call to `_update_buttons()` coordinator
- Implement `_update_storage_cap_button()`: reads config name, max_levels, current level; shows MAX or `"{name} {cur}h → {next}h ({cost} shards)"` with affordability enable/disable
- Implement `_on_storage_cap_pressed()` calling `MetaManager.purchase_gold_storage_cap()`

**[EDITOR]** In `scenes/hub/LabUpgradeScreen.tscn`: add `Button` node named `StorageCapButton` as sibling of `TransmuterButton`; assign to `_storage_cap_button` export.

### Step 6 — GoldDisplay (depends on Step 4, parallel with Step 5)

In `scenes/hub/GoldDisplay.gd`:
- Add `@export var _cap_label: Label`
- In `_ready()`: call `_update_cap_label()` immediately; connect `MetaManager.shards_changed` → `func(_n: int) -> void: _update_cap_label()`
- Implement `_update_cap_label()`: if `MetaManager.is_gold_generator_owned` → `_cap_label.text = "Cap: {n}h".format({"n": MetaManager.gold_storage_cap_hours})`; else `_cap_label.text = ""`

**[EDITOR]** In `scenes/hub/GoldDisplay.tscn`: add `Label` node named `CapLabel` as child; assign to `_cap_label` export.

### Step 7 — Unit tests (parallel with Steps 3–5)

Extend `tests/unit/test_meta_manager_impl_gold.gd`:

**Update existing tests** — all `apply_offline_gold` calls must gain a `cap_seconds` argument. Use `14400` (4h default) for all existing tests so they pass unchanged.

**New tests**:
- `test_apply_offline_gold_over_cap_is_clamped()` — 6h elapsed, 4h cap → 4h gold
- `test_apply_offline_gold_under_cap_is_not_clamped()` — 2h elapsed, 4h cap → 2h gold
- `test_apply_offline_gold_exactly_at_cap()` — 4h elapsed, 4h cap → 4h gold
- `test_apply_offline_gold_updates_timestamp()` — after credit, `gold_last_saved_timestamp == now_unix`
- `test_apply_offline_gold_first_boot_sets_timestamp()` — timestamp=0, after call timestamp == now_unix, gold == 0
- `test_get_gold_storage_cap_seconds_level_0()` — base=4, per_level=4, level=0 → 14400
- `test_get_gold_storage_cap_seconds_level_1()` — base=4, per_level=4, level=1 → 28800
- `test_purchase_gold_storage_cap_success()` — deducts shards, increments level, saves
- `test_purchase_gold_storage_cap_insufficient_shards()` — returns false, no mutation, no save
- `test_purchase_gold_storage_cap_at_max_returns_false()` — returns false, no save
- `test_purchase_gold_storage_cap_idempotent_at_max()` — two purchases at max_levels=1 → second returns false

## Key Constraints

- `apply_offline_gold()` signature change is a **breaking change to all callers and tests** — every call site must add `cap_seconds`.
- `tick_gold()` is untouched — in-session gold accumulation is not capped (FR-009).
- No new autoload or scene is created (existing `LabUpgradeScreen` and `GoldDisplay` are extended).
- All config reads in the autoload use `.get("key", fallback)` — missing config entries must not crash.
