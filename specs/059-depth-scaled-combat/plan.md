# Implementation Plan: Depth-Scaled Combat

**Branch**: `059-depth-scaled-combat` | **Date**: 2026-03-18 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/059-depth-scaled-combat/spec.md`

## Summary

Replace the single global `wave_config` in `dungeon_config.json` with a `depth_tiers` array. Each tier maps a depth range to an initial spawn count + wave sequence. A new `DepthTierConfig` data model wraps each tier entry. `RoomSpawner` caches the tier array at load time, then resolves the correct `WaveConfig` in `_on_player_entered()` (after `depth` has been set by RoomLoader) before spawning begins. All existing wave-spawning logic downstream is unchanged.

## Technical Context

**Language/Version**: GDScript 4.6 (static typing)
**Primary Dependencies**: None — modifies existing data layer and RoomSpawner
**Storage**: `data/dungeon_config.json` — `wave_config` key replaced by `depth_tiers` array
**Testing**: GUT — `DepthTierConfig.from_dict` and `find_for_depth` are pure static logic; GUT unit tests mandatory
**Target Platform**: Android mobile / Windows dev
**Performance Goals**: O(n) over tier count (always ≤ 4) on player entry — negligible
**Constraints**: `depth` is set by RoomLoader *after* `_ready()` fires; tier resolution must happen in `_on_player_entered()`, not `_load_config()`

## Constitution Check

- **I. Single Responsibility** ✅ — `DepthTierConfig` owns tier data parsing. `RoomSpawner` owns spawn orchestration. No cross-concern mixing.
- **II. Data-Driven Content** ✅ — All depth tier values (spawn counts, wave sizes, thresholds) live in `dungeon_config.json`. Zero numeric constants in `.gd` files.
- **III. Mobile-First Performance** ✅ — Tier lookup is a one-time O(n≤4) linear scan on player entry. No per-frame cost.
- **IV. Editor-Centric Workflow** ✅ — No `.tscn` changes. No new Inspector exports required.
- **V. Simplicity & YAGNI** ✅ — `DepthTierConfig` is used by `RoomSpawner` directly; no speculative abstractions. `find_for_depth` has exactly one call site today, but the static pure-logic method is the minimal testable unit for the tier lookup — justified.
- **VI. Early Return** ✅ — `find_for_depth` returns on first match; `_on_player_entered` returns early if no tier found for depth.

## Key Design Decision: Deferred Tier Resolution

`_load_config()` fires in `_ready()`, before RoomLoader sets `depth`. Solution:

1. `_load_config()` reads `depth_tiers` from JSON and stores them as `var _depth_tiers: Array[DepthTierConfig]` directly on `RoomSpawner` (not in `RoomSpawnConfig` — these are global dungeon params, not per-room spawn config).
2. `_on_player_entered()` calls `_resolve_wave_config()` before any spawn logic. This method finds the matching tier for `self.depth`, constructs a `WaveConfig`, and assigns it to `_config.wave_config`.
3. All existing `_spawn_wave` / `_on_enemy_defeated` logic is unchanged — they consume `_config.wave_config` as before.

Elite and Boss rooms are excluded from tier resolution (same guard as current: `not room_type_id.contains("Boss")` and `not room_type_id.contains("Elite")`).

## Depth Tier Table (JSON)

```json
"depth_tiers": [
  { "depth_min": 1, "depth_max": 1, "waves": [3], "trigger_threshold": 2, "alive_cap": 4, "min_spawn_distance": 200.0 },
  { "depth_min": 2, "depth_max": 2, "waves": [4], "trigger_threshold": 2, "alive_cap": 4, "min_spawn_distance": 200.0 },
  { "depth_min": 3, "depth_max": 4, "waves": [4, 2], "trigger_threshold": 2, "alive_cap": 4, "min_spawn_distance": 200.0 },
  { "depth_min": 5, "depth_max": -1, "waves": [4, 2, 1], "trigger_threshold": 2, "alive_cap": 4, "min_spawn_distance": 200.0 }
]
```

`depth_max: -1` means unbounded (depth 5+). The old `wave_config` top-level key is removed.

## Static Return Type Note

`DepthTierConfig.from_dict` and `find_for_depth` MUST declare return type `-> Resource` (not `-> DepthTierConfig`). This matches the fix applied to `WaveConfig.from_dict` in feature 056 — self-referential class_name return types fail compilation in headless GUT mode for files not yet opened in the editor.
