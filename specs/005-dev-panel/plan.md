# Implementation Plan: Dev Panel

**Date**: 2026-02-21 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/005-dev-panel/spec.md`

---

## Summary

A `CanvasLayer`-based dev panel containing four buttons is conditionally instantiated at game start when `const DEV_MODE: bool = true` in `Main.gd`. The panel emits signals; `Main.gd` connects them to `RunManager` calls and stub log messages. When `DEV_MODE` is false, zero panel nodes exist.

Also fixes a pre-existing bug: `RunManager.end_run("dead")` in `Main.gd` passes an unexpected argument to a zero-parameter function.

---

## Technical Context

**Language/Version**: GDScript (Godot 4.6)
**Primary Dependencies**: Godot 4.6 UI system; `RunManager` autoload (004-run-manager)
**Storage**: None — panel is purely in-memory/scene-tree
**Testing**: Manual playtest per `quickstart.md`; Godot Output panel for log validation
**Target Platform**: Android mobile (portrait 1080×1920); Windows for development
**Performance Goals**: Zero overhead when `DEV_MODE = false` (FR-009); panel itself is 4 buttons — negligible cost when active
**Constraints**: Panel must render above all game content (CanvasLayer required); buttons must be touch-friendly (≥44px height)
**Scale/Scope**: 1 scene, 1 script, 1 file change (Main.gd)

---

## Constitution Check

### I. Single Responsibility ✅

- `DevPanel.gd` — one responsibility: render dev buttons and emit signals. Does not reference any autoload.
- `Main.gd` — already the game orchestrator; adding DEV_MODE gating and signal connections is within its existing domain.
- No new autoload. DevPanel is instantiated as a regular scene child.

### II. Data-Driven Content ✅

- No balance values. No JSON changes. The four button labels are UI copy, not tunable content.

### III. Mobile-First Performance ✅

- `CanvasLayer` is a lightweight overlay — no additional draw passes beyond the 4 buttons.
- When `DEV_MODE = false`: zero nodes, zero cost.
- Button sizes set to ≥60px height — comfortably above the 44px touch target minimum.

### IV. Editor-Centric Workflow ✅

- `DevPanel.tscn` is built in the Godot Editor. Not edited as raw text.
- All node hierarchy, anchors, and button text set through the Editor Inspector.

### V. Simplicity & YAGNI ✅

- `DEV_MODE` is a single `const bool` in Main.gd — no config system, no env vars.
- Panel uses runtime `instantiate()` — the simplest mechanism that satisfies FR-009.
- No drag functionality, no collapsible panel, no keyboard shortcuts — none requested.

All decisions resolved. See `research.md` for full rationale.

|---|---|

| Panel node type | `CanvasLayer` (layer=128) — guaranteed above game content |

## Implementation Phases

### Phase 1: Setup — Build DevPanel Scene (Editor Task)

**Purpose**: Scene must exist before a script can be attached or preloaded.

- [ ] T001 In the Godot Editor, create folder `scenes/ui/dev/` and build `DevPanel.tscn`: root node `CanvasLayer` named `DevPanel` (layer=128); add child `PanelContainer`; inside it add `VBoxContainer`; inside that add four `Button` nodes named `StartRun`, `EndRun`, `CashOut`, `StartBoss` with matching text labels; anchor `PanelContainer` to top-left with a small margin (e.g. offset 10px from each edge); set minimum button size to 200×60 in the Inspector — `scenes/ui/dev/DevPanel.tscn`

**Checkpoint**: Scene opens in Editor with no errors; all four buttons visible in the 2D viewport at top-left.

---

### Phase 2: User Story 1 — DEV_MODE Gating (Priority: P1)

**Goal**: Panel appears when `DEV_MODE = true`, zero nodes when `false`.

- [ ] T002 [P] Create `scenes/ui/dev/DevPanel.gd`: `class_name DevPanel extends CanvasLayer`; declare four signals: `signal start_run_pressed`, `signal end_run_pressed`, `signal cash_out_pressed`, `signal start_boss_pressed`; declare `@onready` vars for all four buttons using their node paths; in `_ready()` connect each button's `pressed` signal to emit its corresponding DevPanel signal — `scenes/ui/dev/DevPanel.gd`
- [ ] T003 [P] Update `scenes/core/Main.gd`: add `const DEV_MODE: bool = true` at the top of the file; add `const _DEV_PANEL_SCENE = preload("res://scenes/ui/dev/DevPanel.tscn")`; in `_ready()` add a DEV_MODE guard block that instantiates the panel, calls `add_child()`, and connects all four signals to their handlers (see contracts/dev-panel.md); fix the existing bug `RunManager.end_run("dead")` → `RunManager.end_run()` — `scenes/core/Main.gd`

**Checkpoint**: US1 complete — `DEV_MODE = true`: panel visible with 4 buttons. `DEV_MODE = false`: no panel, no errors.

## Notes

- T001 is an Editor task — must be done before any script work.
- T002 and T003 are fully parallel after T001 (different files).
- The `_DEV_PANEL_SCENE` preload in Main.gd will cause a load error if `DevPanel.tscn` does not exist — complete T001 first.
- `Main.gd` already auto-starts a run in `_ready()` (`RunManager.start_run("endless")`). The DevPanel's Start Run button will restart it — this is correct per spec ("run resets").
- The bug fix in T003 (`end_run("dead")` → `end_run()`) is required regardless of DEV_MODE — it would crash on player death.
