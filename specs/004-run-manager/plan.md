# Implementation Plan: Run Manager

**Date**: 2026-02-21 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/004-run-manager/spec.md`

---

## Summary

Expand `RunManager` from a minimal room-cleared tracker into a full run session manager. Adds run lifecycle (`start_run` / `end_run`), session state (run_id, mode, tier, currency, room tracking), two stub services (DifficultyService, RewardsService), and signals (`run_ended`, `room_cleared`). RoomSpawner is extended with a `room_entered` signal and self-registration so RunManager can track room navigation automatically.

---

## Technical Context

**Language/Version**: GDScript (Godot 4.6)
**Primary Dependencies**: Godot 4.6 engine; existing `RoomSpawner.gd` (003-enemy-spawning); existing `RunManager.gd` autoload
**Storage**: In-memory only — no JSON data changes, no save file writes
**Testing**: Manual playtest per `quickstart.md`; Godot Output panel for validation
**Target Platform**: Android mobile (portrait 1080×1920); Windows for development
**Performance Goals**: All session reads/writes are O(1) — no performance concern
**Constraints**: Session fields must update within the same frame as their triggering event (SC-001, SC-002)
**Scale/Scope**: One run at a time; up to 100+ rooms per run (SC-005)

## Constitution Check

### I. Single Responsibility ✅

- `RunManager` domain is "run session management" — lifecycle, room tracking, currency, and service access all fall within this single domain. No new domain is absorbed.
- `DifficultyService.gd` — single responsibility: provide difficulty scaling value.
- `RewardsService.gd` — single responsibility: provide room reward data.
- `RoomSpawner.gd` extension adds one signal and one method call — responsibility unchanged.
- No new autoloads. Services are owned by RunManager, not independently autoloaded.

### II. Data-Driven Content ✅

- No balance values in code. `current_tier` is stored as data, not computed.
- No JSON files added or modified — this feature is pure logic/state.
- Service stubs return constants (1.0, {}); when real logic is added, it will read from JSON data files.

### III. Mobile-First Performance ✅

- All operations are O(1) dictionary and variable assignments.
- No per-frame processing added to RunManager.
- No shaders, physics, or rendering changes.

### IV. Editor-Centric Workflow ✅

- No scene (`.tscn`) changes required. RunManager is script-only.
- `RoomSpawner.gd` changes are script-only (signal addition + method call).
- No Inspector properties added.

### V. Simplicity & YAGNI ⚠️ Justified Exception

- **DifficultyService and RewardsService stubs**: The constitution prohibits "stub scripts acting as future hooks." These stubs are a justified exception: they define a **callable interface used today** by RunManager and any caller querying `RunManager.difficulty_service`. The stub IS the current implementation (returns real values: 1.0 and {}). The future feature replaces the logic, not the interface. Explicitly requested in the spec.
- No other abstractions introduced. No base classes. No wave hooks or placeholder infrastructure beyond the two stubs above.

---

### Phase 3: Core — Rewrite RunManager

**Purpose**: Full session state, lifecycle, room tracking, currency, and service wiring in one file.

- [ ] T004 Rewrite `autoload/RunManager.gd` with the following, preserving existing `mark_room_cleared` / `is_room_cleared` methods:
  - Signals: `signal run_ended`, `signal room_cleared(room_id: String)`
  - Session vars (all typed): `var run_id: String = ""`, `var is_run_active: bool = false`, `var run_mode: String = ""`, `var current_tier: int = 1`, `var run_start_time: float = 0.0`, `var run_currency: float = 0.0`, `var current_room: Node = null`, `var current_room_index: int = 0`; preserve `var cleared_rooms: Dictionary = {}`
  - Service vars: `var difficulty_service: DifficultyService`, `var rewards_service: RewardsService`
  - `_ready()`: instantiate `difficulty_service = DifficultyService.new()` and `rewards_service = RewardsService.new()`
  - `start_run(mode: String) -> void`: validate mode (push_warning if not "endless"/"boss"), set all session fields to initial values, set `run_id = str(Time.get_ticks_msec())`, set `is_run_active = true`
  - `end_run() -> void`: guard `if not is_run_active: return`; set `is_run_active = false`; emit `run_ended`
  - `register_room(spawner: Node) -> void`: connect `spawner.room_entered` to `_on_room_entered.bind(spawner)`; connect `spawner.room_cleared` to `_on_room_cleared`
  - `add_currency(amount: float) -> void`: guard `if not is_run_active: push_warning(...); return`; set `run_currency = maxf(run_currency + amount, 0.0)`
  - `_on_room_entered(room_id: String, spawner: Node) -> void`: guard `if not is_run_active: return`; set `current_room = spawner`; increment `current_room_index`
  - `_on_room_cleared(room_id: String) -> void`: guard `if not is_run_active: return`; call `mark_room_cleared(room_id)`; emit `room_cleared(room_id)`
  - preserve `start_new_run()` (calls `start_run("endless")` for backward compat, or keeps existing reset logic)
  - `autoload/RunManager.gd`

**Checkpoint**: Run game — `[RoomSpawner] ready` lines appear with no errors. Enter CombatRoom01 — `current_room_index` is 1. Defeat enemies — `room_cleared` fires.

## Notes

- T004 is the largest task — it rewrites a file. Write the complete new file in one pass; do not attempt incremental edits to the existing RunManager.
- `register_room` uses `.bind(spawner)` to thread the spawner reference into `_on_room_entered` since the `room_entered` signal only carries `room_id`.
- `start_new_run()` is kept for backward compatibility (RoomSpawner currently calls `RunManager.mark_room_cleared` not `start_new_run`, so there may be no active callers — confirm before removing).
- All debug print logging from RoomSpawner is preserved as-is; no new logging required in RunManager unless helpful during T006 validation.
