# Implementation Plan: Essence Currency

**Branch**: `014-essence-currency` | **Date**: 2026-02-28 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/014-essence-currency/spec.md`

## Summary

Add a run currency called "essence". Enemies drop essence on death; the amount is
`floori(base_essence × (1 + 0.10 × depth))` where `base_essence` is per-enemy-type
data in `enemies.json` and `depth` is the room's grid distance from the start room.
When a run ends, essence is cashed out: 100% on CASH_OUT, 85% (floored) on DIED.
Cash-out is a print statement for now; no persistent wallet.

## Technical Context

**Language/Version**: GDScript 4 (Godot 4.6)
**Primary Dependencies**: Godot 4.6 engine — no third-party libraries
**Storage**: `data/enemies.json` extended with `base_essence` per enemy entry
**Testing**: Manual validation via Godot Output panel (12 scenarios in quickstart.md)
**Target Platform**: Mobile (Android/iOS); Windows editor
**Project Type**: Single Godot project
**Performance Goals**: Simple float arithmetic per kill — no performance concern
**Constraints**: None — all calculation happens in response to discrete kill events
**Scale/Scope**: 6 file modifications; 0 new files; 0 new scenes

## Constitution Check

- **I. Single Responsibility** ✅ — EnemyData gains one data field. Enemy gains one
  getter. RoomSpawner gains one export field, one signal, and reports kill facts only —
  no currency logic. RoomLoader gains one assignment line. RunManager gains the formula
  handler and cash-out block — currency logic stays in the run-session manager.
- **II. Data-Driven Content** ✅ — `base_essence` per enemy lives in `enemies.json`.
  The depth formula constants (`0.10`, `0.85`) are game-balance values; see Complexity
  Tracking below for justification.
- **III. Mobile-First Performance** ✅ — `floori()` and a single `add_currency()` call
  per kill event. Negligible cost.
- **IV. Editor-Centric Workflow** ✅ — No scene files modified. `depth` is set
  programmatically by RoomLoader (same pattern as `difficulty_mult`).
- **V. Simplicity & YAGNI** ✅ — No persistent wallet, no UI, no spending system.
  Cash-out is a print stub. All deferred to future features.

*Post-design re-check*: All constraints satisfied. No violations.

## Implementation Phases

### Phase 1 — Data Layer

**File**: `data/enemies.json`

Add `"base_essence": 2` to slime, `"base_essence": 3` to skeleton.

**File**: `scripts/data_models/EnemyData.gd`

Add field:
```gdscript
var base_essence: float
```

In `from_dict()`, add after the existing field assignments:
```gdscript
d.base_essence = float(data.get("base_essence", 0.0))
```

No assert — missing key silently defaults to 0.0 (FR-008).

### Phase 2 — ResourceManager: base_essence lookup

**File**: `autoload/ResourceManager.gd`

Extend the existing enemy data load to also cache `base_essence` per enemy ID.
Add a new `_enemy_essence_cache: Dictionary = {}` field and populate it inside
the existing `_load_enemy_ids()` (rename to `_load_enemy_data()` to reflect broader
scope):

```gdscript
var _enemy_essence_cache: Dictionary = {}

func get_enemy_base_essence(id: String) -> float:
    if not _enemy_ids_loaded:
        _load_enemy_data()
    return _enemy_essence_cache.get(id, 0.0)
```

Inside `_load_enemy_data()`, alongside the existing ID append:
```gdscript
_enemy_essence_cache[entry["id"]] = float(entry.get("base_essence", 0.0))
```

Missing `base_essence` defaults to `0.0` — no crash (FR-008).

### Phase 3 — RoomSpawner: depth field + enemy_defeated signal

**File**: `scripts/dungeon/RoomSpawner.gd`

RoomSpawner reports what died — nothing more. Currency math belongs in RunManager.

Add export field alongside `difficulty_mult`:
```gdscript
@export var depth: int = 0
```

Add new signal:
```gdscript
## Emitted for every enemy defeat. Carries only the enemy type — consumers
## resolve stats and context from their own sources.
signal enemy_defeated(enemy_type_id: String)
```

In `_spawn_enemies()`, the connection is unchanged in style but now binds the
enemy type ID:
```gdscript
enemy.defeated.connect(_on_enemy_defeated.bind(sp.enemy_id))
```

Update `_on_enemy_defeated` — emit the new signal, no currency or stats logic:
```gdscript
func _on_enemy_defeated(enemy_type_id: String) -> void:
    enemy_defeated.emit(enemy_type_id)
    _living_count -= 1
    print("[RoomSpawner] enemy defeated — remaining: {count}".format({"count": _living_count}))
    if _living_count == 0:
        RunManager.mark_room_cleared(room_id)
        room_cleared.emit(room_id)
        print("[RoomSpawner] cleared room_id='{id}' room_type='{type}'".format({"id": room_id, "type": room_type_id}))
```

Note: `get_base_essence()` getter on Enemy is no longer needed and should NOT be
added.

### Phase 4 — RoomLoader: set depth on spawner

**File**: `scripts/dungeon/RoomLoader.gd`

Add one line after `spawner.difficulty_mult = room_mult`:
```gdscript
spawner.depth = _dungeon_gen.rooms_by_id[room_id].get("depth", 0)
```

### Phase 5 — RunManager: session depth + connect enemy_defeated + cash-out

**File**: `scripts/managers/RunManager.gd`

Add session field alongside other session state:
```gdscript
var current_room_depth: int = 0
```

Reset it in `start_run()` alongside the other resets:
```gdscript
current_room_depth = 0
```

Cache depth when a room is entered — extend `_on_room_entered()`:
```gdscript
func _on_room_entered(room_id: String, spawner: Node) -> void:
    ...existing lines...
    current_room_depth = (spawner as RoomSpawner).depth
```

In `register_room()`, add connection to `enemy_defeated` alongside the existing two:
```gdscript
spawner.enemy_defeated.connect(_on_enemy_defeated)
```

Add new handler — owns the formula, reads its own session state:
```gdscript
func _on_enemy_defeated(enemy_type_id: String) -> void:
    var base_essence: float = ResourceManager.get_enemy_base_essence(enemy_type_id)
    var essence: int = floori(base_essence * (1.0 + 0.10 * float(current_room_depth)))
    if essence > 0:
        add_currency(float(essence))
```

Insert cash-out block in `end_run()` after `is_run_active = false`, before
`run_ended.emit(reason)`:
```gdscript
var cashed_out: int
if reason == EndReason.DIED:
    cashed_out = floori(run_currency * 0.85)
else:
    cashed_out = floori(run_currency)
print("[Essence] {amount} essence cashed out".format({"amount": cashed_out}))
```

### Phase 6 — Validation

Run all 12 scenarios in `specs/014-essence-currency/quickstart.md`.
Pay special attention to Scenario 7 (DIED cash-out = 85%) and Scenario 10
(essence resets between runs).

## State Lifecycle

```
Enemy killed:
  StatsComponent.died → Enemy._on_died() → Enemy.defeated.emit()
  → RoomSpawner._on_enemy_defeated(enemy_type_id)
      enemy_defeated.emit(enemy_type_id)          ← reports type only
      _living_count -= 1
      if _living_count == 0: room_cleared logic

  → RunManager._on_enemy_defeated(enemy_type_id)  ← owns formula + context
      base_essence = ResourceManager.get_enemy_base_essence(enemy_type_id)
      depth        = current_room_depth            ← cached at room entry
      essence      = floori(base_essence × (1 + 0.10 × depth))
      if essence > 0: add_currency(essence)

Room entered:
  RunManager._on_room_entered(room_id, spawner)
      current_room_depth = (spawner as RoomSpawner).depth  ← cached here

Run ends (end_run(reason)):
  is_run_active = false
  cashed_out = floori(run_currency × 0.85)  [DIED]
               OR floori(run_currency)        [CASH_OUT]
  print "[Essence] X essence cashed out"
  run_ended.emit(reason)

Next run starts (start_run()):
  run_currency = 0.0  ← resets here
```

## Complexity Tracking

| "Violation" | Why Needed | Simpler Alternative Rejected Because |
|---|---|---|
| `0.10` depth multiplier constant in RoomSpawner.gd | Formula constant — identical pattern to `difficulty_mult = 1.0 + 0.12 × depth` already in DungeonGenerator. Moving to JSON would require a new schema entry for a single scalar. | Over-engineering for one formula constant. |
| `0.85` death penalty in RunManager.gd | Single scalar affecting end_run behaviour. No other system needs to read it. | Adding a config entry for one value used in one place is premature. |
