# Implementation Plan: Hub Room

**Branch**: `013-hub-room` | **Date**: 2026-02-27 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/013-hub-room/spec.md`

## Summary

Add a hub room as the game's entry point. The player starts in the hub (not directly in a dungeon). A `TeleportDoor` placeholder scene contains a `Button` labelled "Teleport". Pressing the button starts the run and removes the hub from the scene. `Main.gd` is modified to instantiate the hub at startup instead of immediately calling `start_run()`.

**Activation model**: Button press (not proximity). `Button.pressed` → `TeleportDoor.teleport_activated` → `HubRoom.hub_exited` → `Main._on_hub_exited()` → `RunManager.start_run("endless")`.

## Technical Context

**Language/Version**: GDScript 4 (Godot 4.6)
**Primary Dependencies**: Godot 4.6 engine — no third-party libraries
**Storage**: None — hub has no persistent data
**Testing**: Manual validation via Godot Editor viewport, Remote Inspector, and Output panel (13 scenarios in quickstart.md)
**Target Platform**: Mobile (Android/iOS); Windows editor
**Project Type**: Single Godot project
**Performance Goals**: Hub is a single-room scene with zero enemies — no performance concern
**Constraints**: `Button.pressed` comes from the UI input system (not physics) — no `call_deferred` needed
**Scale/Scope**: 2 new scenes (Editor), 2 new scripts, 1 modified script (Main.gd)

## Constitution Check

- **I. Single Responsibility** ✅ — `TeleportDoor.gd`: detects button press and emits signal only. `HubRoom.gd`: owns hub room lifecycle only. `Main.gd` modification: minimal (2 lines removed, hub setup + handler added).
- **II. Data-Driven Content** ✅ — Hub has no balance data. No JSON needed. Structural scene, not content.
- **III. Mobile-First Performance** ✅ — One `Button` (Control) + one `ColorRect` + one `Node2D`. Negligible cost. Godot `Button` handles mobile touch automatically.
- **IV. Editor-Centric Workflow** ✅ — `HubRoom.tscn` and `TeleportDoor.tscn` MUST be created in Godot Editor. Scripts authored in code editor and attached via Editor.
- **V. Simplicity & YAGNI** ✅ — No future-proofing for meta progression. Hub is a plain Node2D room. Button placeholder explicitly declared. No premature abstractions.

*Post-design re-check*: All constraints satisfied. No violations.

## Implementation Phases

### Phase 1 — TeleportDoor.gd (new script)

**File**: `scenes/hub/TeleportDoor.gd`

```gdscript
class_name TeleportDoor
extends Node2D

signal teleport_activated


func _ready() -> void:
    $Button.pressed.connect(_on_button_pressed)


func _on_button_pressed() -> void:
    if not RunManager.is_run_active:
        teleport_activated.emit()
```

`Button.pressed` fires from Godot's UI input system — no physics flush, no `call_deferred` needed.

### Phase 2 — HubRoom.gd (new script)

**File**: `scenes/hub/HubRoom.gd`

```gdscript
class_name HubRoom
extends Node2D

signal hub_exited


func _ready() -> void:
    var teleport: TeleportDoor = get_node("TeleportDoor")
    teleport.teleport_activated.connect(_on_teleport_activated)


func _on_teleport_activated() -> void:
    hub_exited.emit()
    queue_free()
```

`hub_exited` fires synchronously before `queue_free()`. Main's handler runs and starts the run while the node is still valid.

### Phase 3 — TeleportDoor.tscn (Godot Editor — required)

**Scene**: `scenes/hub/TeleportDoor.tscn`

Editor steps:
1. Create new scene with root type `Node2D`, rename root to `TeleportDoor`.
2. Add child `ColorRect` as visual placeholder (e.g. 120×180 px, dark colour) — this represents the "door" shape.
3. Add child `Button`, set `text = "Teleport"`, size and position it over or near the ColorRect.
4. Attach script `TeleportDoor.gd` to the root node.
5. Save as `scenes/hub/TeleportDoor.tscn`.

### Phase 4 — HubRoom.tscn (Godot Editor — required)

**Scene**: `scenes/hub/HubRoom.tscn`

Editor steps:
1. Create new scene with root type `Node2D`, rename root to `HubRoom`.
2. Add child `ColorRect`, set size 1920×1080, background colour (e.g. dark grey `#2a2a3a`).
3. Add child instance of `TeleportDoor.tscn`, name it `TeleportDoor`, position it near player start (e.g. (0, -200)).
4. Attach script `HubRoom.gd` to the root node.
5. Save as `scenes/hub/HubRoom.tscn`.

### Phase 5 — Main.gd modification

**File**: `scenes/core/Main.gd`

Changes:
1. Add preload constant at top: `const _HUB_ROOM_SCENE = preload("res://scenes/hub/HubRoom.tscn")`
2. Add field: `var _hub_room: Node = null`
3. In `_ready()`, replace lines 21–22 with:
   ```gdscript
   _hub_room = _HUB_ROOM_SCENE.instantiate()
   add_child(_hub_room)
   _hub_room.hub_exited.connect(_on_hub_exited)
   ```
4. Add new method:
   ```gdscript
   func _on_hub_exited() -> void:
       RunManager.start_run("endless")
       GlobalSignals.gameplay_started.emit()
   ```

### Phase 6 — Validation

Run all 13 scenarios in `specs/013-hub-room/quickstart.md`. Pay special attention to Scenario 5 (proximity alone does NOT start run) and Scenario 11 (double-press guard).

## State Lifecycle

```
Game launch:
  Main._ready()
    ├── DevPanel setup (dev mode only)
    ├── Wire joystick → MovementComponent
    ├── Connect player.died → _on_player_died
    ├── Instantiate HubRoom → add_child → connect hub_exited
    └── (no start_run, no gameplay_started)

While in hub:
  Player moves freely via joystick
  "Teleport" Button always visible
  RunManager.is_run_active == false
  ExplorationHUD hidden

Player presses Teleport button:
  Button.pressed → TeleportDoor._on_button_pressed()
  → !is_run_active → teleport_activated.emit()
  → HubRoom._on_teleport_activated()
  → hub_exited.emit()       ← Main._on_hub_exited() called synchronously
  → queue_free()            ← hub removed at end of frame

Main._on_hub_exited():
  RunManager.start_run("endless")      ← triggers DungeonGenerator
  GlobalSignals.gameplay_started.emit() ← shows ExplorationHUD

DungeonGenerator → RoomLoader → player placed in StartRoom01
```
